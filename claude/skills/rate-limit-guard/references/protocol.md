# 待機・再開プロトコル詳細

`SKILL.md` の「待つ / 停止 / 再開」を実行する際の具体手順とエッジケース。

## チェックポイントの置き場所と書式

待機・停止の前にチェックポイントを **セッション別ファイル** として残す:

```bash
mkdir -p ~/.claude/rate-limit-guard/checkpoints
# 保存先: ~/.claude/rate-limit-guard/checkpoints/${CLAUDE_CODE_SESSION_ID}.md
```

セッション別にする理由: 複数の Claude セッションが並行作業しているとき、固定パス1つだと互いのチェックポイントを上書きして再開情報を失う。`CLAUDE_CODE_SESSION_ID`(UUID)は Bash 環境変数として参照でき、PreToolUse フックが見る session_id とも通常一致する(万一フックのヒントが示すパスと自分の環境変数がズレていても、下記の `ls -t` フォールバックで見つかるので致命的ではない)。未設定の環境では、タスク内容のkebab-caseスラッグ + 日時(例: `auth-refactor-20260612-1130.md`)をファイル名にフォールバックする。

- **同一セッション内での再開**(待機明け): 自分のIDのファイルを読む。再開が完了して作業に戻ったら、そのファイルは削除してよい(残すと古いチェックポイントが堆積する)。
- **別セッションからの再開**(セッションを閉じて後で再開、または `wrap_up_and_stop` 後): 新しいセッションはIDが変わるため自分のIDでは見つからない。`ls -t ~/.claude/rate-limit-guard/checkpoints/` で更新順に並べ、ヘッダのタスク名・保存時刻で自分が引き継ぐべきものを特定する。だからヘッダにタスク名と時刻を必ず書く。

次のセッション/起床がこのファイルだけ読めば再開できる粒度にする:

```markdown
# Checkpoint — <タスク名>  (saved: <人間可読な時刻>, session: <CLAUDE_CODE_SESSION_ID>)

## ゴール
<このタスク全体で達成したいこと>

## 完了したこと
- <箇条書き>

## 次にやること(再開時の最初の一手)
1. <具体的な次アクション>

## 触ったファイル / WIPコミット
- <ファイル一覧、または `git log -1` のハッシュ>

## チーム状態(チーム作業中のみ)
| メンバー | 割り当て | 完了 | 残り |
|---|---|---|---|
| <名前> | <タスク> | <済んだこと> | <再投入する作業> |

## 待機/停止理由
- action: wrap_up_and_wait, limiting_window: five_hour, used_percentage: 92, wait_until: <時刻>
```

可能なら WIP コミットも併用する(最も確実な復元手段)。コミットメッセージは日本語で。

## 待機の仕方(`wrap_up_and_wait`)

チェック結果の `wait_until`(エポック秒)まで待つ。リセット判定は **現在時刻 ≥ `wait_until`** で行う。待機中は API 呼び出しが無く `used_percentage` が更新されないため、使用率の低下を待ってはいけない(リセット到達後の再チェックでは、チェッカーが `window_elapsed: true` として古い使用率を超過扱いから外す)。

なおチェッカーは、超過していても該当枠の `resets_at` が取れない場合は `wrap_up_and_wait` ではなく `action: unknown`(note に理由)を返す — 待機目標を計算できないまま当て推量で待たないため。その場合は切り上げまで行い、ユーザーに確認する。

`limiting_window` が `seven_day` の場合も手順は同じ(リセットが `RL_MAX_WAIT_SECONDS` 以内と判定済みなので待ってよい)。両枠超過時の `wait_until` は遅い方に合わせてある。

自動再開はセッションが開いている限り `/loop` 不要で動く。優先順:

### 1. 背景待機 + 自動再開(推奨・通常セッションで動く)

`Bash` を `run_in_background: true` で実行し、`wait_until` まで待つ until ループを回す。背景タスクは終了時に自動で再起動をかけるので、その通知で再開する。

```bash
target=$(( WAIT_UNTIL + 60 ))           # WAIT_UNTIL = check結果の wait_until(エポック秒)
until [ "$(date +%s)" -ge "$target" ]; do sleep 30; done
echo "RESET_REACHED"
```

- 前景 `sleep` はハーネスでブロックされるが、`run_in_background: true` の中の `sleep` は動く(実測確認済み)。
- 起こされたら「再開時のチェック」へ。**まだ `wait_until` 未到達なら同じ背景待機を張り直す**(背景タスクが途中で打ち切られても、これで確実に到達まで粘れる)。
- 1本の背景待機で済むので、キャッシュ再構築は実質1回。

### 2. Monitor(進捗表示・長時間向けの代替)

残り時間のハートビートを見たい、または待機が長くて背景待機の張り直しを避けたい場合:

```bash
target=$(( WAIT_UNTIL + 60 ))
while [ "$(date +%s)" -lt "$target" ]; do
  echo "waiting... $(( (target - $(date +%s)) / 60 ))min left"; sleep 60
done
echo "RESET_REACHED"
```

`Monitor` の `timeout_ms` は最大3600000(1時間)。それより長い待機は `persistent: true`(TaskStop か到達で終了)。各行が通知になるのでハートビート間隔は粗め(60s〜)にする。

### 3. `/loop` 自走モードなら ScheduleWakeup でも可

ループで回している場合は各起床で `check-rate-limit.sh` を実行し、`action` に応じて `ScheduleWakeup(min(残り秒数 + 60, 3600))` で次の起床を刻む(`delaySeconds` は [60,3600] クランプなので残り1時間超は1時間刻み)。

### セッションを閉じる場合

背景待機・Monitor・`/loop` いずれもセッションに紐づくので、閉じると止まる。完全に閉じる前提なら `CronCreate`(`recurring:false`, `durable:true`)で `wait_until + 60秒` に再開プロンプトを仕込む(REPL がアイドルかつ Claude 起動中に発火。プロンプトにチェックポイントのフルパスを含めること — 再開セッションはIDが違うので自動では見つけられない)か、チェックポイントを残して次回起動時に再開する運用にフォールバックする。設定変更を伴う場合はユーザー確認のうえで。

## 停止の仕方(`wrap_up_and_stop`・週次枠のリセットが遠い)

待っても枠が戻らないので待機ループは張らない。やることは「安全化 → 報告 → 停止」:

1. SKILL.md の「切りの良いところまで進める」を実行(チーム作業中はメンバー切り上げを含む)。
2. ユーザーへの報告に含める: 週次枠の使用率 / `resets_at` の人間可読時刻(`date -r <epoch>` で変換) / ここまでの進捗サマリ / チェックポイントの**フルパス**(再開セッションはIDが違うため、パスを明示しないと探させることになる) / 再開手順の一行(例: 「リセット後にこのチェックポイントを読んで再開して」)。
3. `PushNotification` が使えるなら短い通知も送る(ユーザーが離席している夜間実行で特に有効)。
4. 停止する。「あと少しだけ」と続行しない — 5時間枠と違い、消費した分は数日戻らない。

リセットが近づいた時点での再開を仕込みたい場合のみ、ユーザーに確認のうえ `CronCreate`(`recurring:false`, `durable:true`)を使う。既定は仕込まず停止。

## チーム切り上げの詳細(リード向け)

`wrap_up_and_wait` / `wrap_up_and_stop` でチームメンバーが走っている場合、**自分の待機・停止より先に**メンバーを畳む。順序を守らないと、待機中もメンバーが同じ枠を消費し続ける。

1. **一斉指示** — 全メンバーに同文を `SendMessage`:
   > 使用量が上限に近づいたため切り上げる。いま着手中の最小単位だけ完了させ、新しいファイル・新しいサブタスクには着手しないこと。終わったら「完了したこと / 未完了 / 触ったファイル / 再開時の注意」を報告して待機せよ。
2. **回収** — 全員の報告を待つ。一定時間(目安5分)応答がないメンバーは状況不明として checkpoint に「応答なし」と記録し、先に進む(報告待ちで枠を使い切っては本末転倒)。
3. **チームチェックポイント** — 上記書式の「チーム状態」表に全員分を記録。
4. **再開** — `wrap_up_and_wait` なら起床後、checkpoint の表に従って各メンバーへ残作業を `SendMessage` で再投入(終了済みメンバーは同じ担当範囲で再スポーン)。`wrap_up_and_stop` ならメンバーをアイドルのまま残さず、シャットダウンを指示してからユーザーに報告する。

メンバー自身がフックのヒントを受けた場合の正しい挙動は「切り上げて進捗をリードに報告し、指示を待つ」まで。メンバーは自分で待機ループを張らない(待機の主導権はリードに一本化する)。

## 再開時のチェック

起床/再開したら、作業に戻る前に必ず:

1. `check-rate-limit.sh` を再実行。
2. `action` が `continue` / `approaching` に戻っていることを確認(まだ超過なら再待機)。`approaching` なら以後毎節目チェックに切り替えて再開してよい。
3. `source` と `fresh` を確認。`fresh: false` でも `window_elapsed: true` が出ていればリセット到達済みなので再開してよい(最初の API 応答で状態ファイルが更新され、次のチェックから実値に戻る)。
4. 自セッションのチェックポイント(`checkpoints/${CLAUDE_CODE_SESSION_ID}.md`)を読み、「次にやること」から再開。チームを畳んでいた場合はメンバーへ再投入。作業に復帰できたらチェックポイントファイルを削除する(堆積防止)。

## 閾値とチューニング

- `RL_THRESHOLD`(既定90): 何%で発動するか。早めに切り上げたいなら 85 など。
- `RL_WARN_THRESHOLD`(既定80): `approaching`(毎節目チェックに切り替える警告域)の下限。
- `RL_MAX_WAIT_SECONDS`(既定21600=6時間): 自動待機する最大秒数。週次枠超過の「待つ/止まる」分岐に使う。夜間実行で朝まで粘ってよいなら 43200(12時間)などに上げる。
- `RL_STALE_SECONDS`(既定180): 状態ファイルを「古い」とみなす秒数。
- これらは `check-rate-limit.sh` を呼ぶ際の環境変数で渡す。
