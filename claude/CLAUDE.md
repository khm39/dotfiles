# Global Rules

- 複数タスクを並列化できる場合は積極的にTeamCreate でエージェントチームを構成し、チームメンバーに作業を分担させること

## Git 操作

- gitのコミットは作業の区切りで適宜行うこと。論理的にまとまった単位（機能追加・バグ修正・リファクタリング等）ごとに分けてコミットする
- 破壊的な操作（force push、reset --hard、ブランチ削除など）はユーザーに確認してから実行する
- コミットメッセージは日本語で書くこと

## コーディング

- コードには原則コメントを書かない。書くのは WHY が非自明な場合（隠れた制約・微妙な不変条件・バグ回避・読み手が驚く挙動）のみ

## エージェントチーム運用

- チームリードは実装を自分で行わず、タスクの分解・割り当て・調整に専念すること
- コストよりもタスクに対して適切なチームサイズとメンバー構成を優先する
- チームメイト1人あたり5〜6タスクが適切な粒度
- スポーン前にタスク分解を十分に行う（計画80%、実行20%）
- チームメイトは会話履歴を引き継がないため、スポーンプロンプトに必要なコンテキストをすべて含める
- 各チームメイトが異なるファイルセットを担当するよう分割し、同一ファイルの同時編集を避ける
- 同一ファイルへの編集が必要な作業や、依存関係が多い逐次処理にはチームを使わない
- 大きな実装タスクでは、レビュワー役のチームメイトを含めることを検討する

## Worktree運用

複数のClaudeセッションが同一リポジトリで並行作業すると、git作業ツリーやインデックスが衝突する。これを避けるため、各セッションは自分専用のworktreeで作業する。

- **発動タイミング**: 編集系ツール（Edit / Write / NotebookEdit）を**最初に呼ぶ直前**に `EnterWorktree` を呼ぶ。コマンドや指示を受けた瞬間ではなく、実際に編集を始める段階で発動する
- **読み取り専用セッションでは呼ばない**: 質問への回答・コード調査・説明のみで完結するセッションでは worktree を作らない
- **二重呼び出しの回避**: 現在のCWDが `.claude/worktrees/` を含む、または `git worktree list` で現在地が登録済みworktreeなら、再度呼ばない
- **命名**: タスク内容を反映した短いkebab-case（例: `fix-login-bug`, `add-worktree-rule`）。決められない場合は `name` 省略で自動生成に任せる
- **base ref**: `worktree.baseRef: fresh`（origin/<default>派生）を settings.json で明示設定済み。未コミット変更を引き継ぐ必要があるときは、ユーザーに確認してから対応する（`git stash` で持ち込む、対象ファイルだけコピーする、など）
- **TeamCreate との併用**: メインがworktree内にいるときに `TeamCreate` でメンバーを立てる場合、メンバー側も `Agent` の `isolation: "worktree"` を指定する。二重worktreeになるが、メンバー作業の隔離は保たれる
- **完了処理**: 作業完了時（PR作成後やタスク終了時）に、ユーザーへ `ExitWorktree` の実行可否を確認する。残作業がありそうなら `action: "keep"`、不要なら `action: "remove"`
