# Global Rules

このファイルは毎ターン全文がコンテキストに載る。記述は短く具体的に保ち、ハーネス既定で十分なことは書かない。

## 作業の進め方

- スコープを守る。依頼の範囲に集中し、気づいた別の改善は勝手に着手せず提案にとどめる
- 完了報告はテスト／実行で動作を確認してから行う。確認できない場合は「未確認」と明示する
- 非自明・判断の分岐がある作業はまず計画を固める（認識合わせは `design-dialogue`、実装方針は `Plan` エージェント）。可逆で小さい作業は確認せず進める
- 長時間・大量編集・自律実行を始める前に `rate-limit-guard` で使用量監視を組み込む

## Git 操作

- 論理的にまとまった単位（機能追加・バグ修正・リファクタリング等）ごとに、作業の区切りでコミットする
- コミットメッセージは日本語で書く
- force push / reset --hard / ブランチ削除などの破壊的操作はユーザーに確認してから実行する

## コーディング

- コードには原則コメントを書かない。書くのは WHY が非自明なとき（隠れた制約・微妙な不変条件・バグ回避・読み手が驚く挙動）のみ
- 既存コードのスタイル・命名・イディオムに合わせる

## 並列実行・エージェントチーム

- 独立した複数タスクは積極的に並列化する。1メッセージで複数ツールを同時に呼び出し、規模が大きければ `TeamCreate` でチームを組む
- リードは実装せず、分解・割り当て・調整に専念する（計画80%／実行20%。スポーン前に分解を終える）
- チームサイズはコストよりタスク適合を優先する。1メンバーあたり5〜6タスクを目安にする
- メンバーは会話履歴を引き継がないため、スポーンプロンプトに必要なコンテキストを全て含める
- メンバーごとに担当ファイルを分け、同一ファイルの同時編集を避ける
- 同一ファイルの編集や依存の強い逐次処理にはチームを使わない
- 大きな実装ではレビュワー役のメンバーを含めることを検討する

## Worktree 運用

複数のClaudeセッションが同一リポジトリで並行作業すると、作業ツリーやインデックスが衝突する。各セッションは専用のworktreeで作業する。

- **発動タイミング**: 編集系ツール（Edit / Write / NotebookEdit）を最初に呼ぶ直前に `EnterWorktree` を呼ぶ。指示を受けた瞬間ではなく、実際に編集を始める段階で発動する
- **読み取り専用では呼ばない**: 回答・調査・説明のみで完結するセッションでは作らない
- **二重呼び出しの回避**: CWDが `.claude/worktrees/` を含む、または `git worktree list` で現在地が登録済みworktreeなら、再度呼ばない
- **命名**: worktree のディレクトリ名とブランチ名は別物。push するブランチ名には `worktree` 等の接頭辞を付けず、タスクを反映した短いkebab-case にする（例 `fix-login-bug`）。決められなければ `name` 省略で自動生成に任せる
- **未コミット変更の引き継ぎ**: 必要時はユーザーに確認してから対応する（`git stash` で持ち込む、対象ファイルだけコピーする、など）
- **TeamCreate との併用**: メインがworktree内のとき `TeamCreate` でメンバーを立てる場合、メンバー側も `Agent` の `isolation: "worktree"` を指定する
- **完了処理**: 作業完了時（PR作成後やタスク終了時）に `ExitWorktree` の可否をユーザーに確認する。残作業がありそうなら `action: "keep"`、不要なら `action: "remove"`
