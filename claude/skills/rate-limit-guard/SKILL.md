# Rate Limit Guard

長時間の作業中に Claude のサブスク使用量を監視し、上限(既定90%)に近づいたら**作業を壊さずに切り上げ → リセットまで待機 → 自動再開**するためのスキル。

「途中でいきなり制限に当たって中途半端な状態で止まる」事故を防ぐのが目的。だから *いつ止まっても安全な状態を作ってから* 待つ、という順序が最重要。

## 対象範囲

**このスキルが見るのは 5時間枠(セッション枠)のみ。** 週次(7日)枠は対象外とする。理由は、週次枠のリセットは最大で数日先で、自動待機しても割に合わず(キャッシュも無駄)、本スキルの「切り上げ→待機→自動再開」という設計とかみ合わないため。状態ファイルに `seven_day` があっても無視する。週次枠が高くても5時間枠に余裕があれば作業は続行してよい。

## 仕組み(独立性)

使用量の**真の値**はサブスクの 5時間枠の使用率で、これは statusLine に流れてくる `rate_limits` ストリームからしか取れない。本スキルは既存の statusline スクリプトに**小さなキャプチャ行を追記**して独立した状態ファイルを生成する。**statusLine のコマンド自体は差し替えない**(表示・動作は不変)。生成するファイルは本スキル専用で、外部ツール(vibe-island 等)の出力には依存しない。

- **状態キャプチャ(追記)** — `scripts/setup.sh` が既存の `~/.claude/statusline.sh` に、`rate_limits` を独自ファイル `~/.claude/rate-limit-guard/state.json` へ書き出すマーカー付きブロックを**追記**する。`uninstall` で正確に除去でき、`install`/`uninstall` の往復で原本にバイト一致で戻る。
- **チェッカー** `scripts/check-rate-limit.sh` — 上記の状態ファイルを読み、5時間枠の使用率・リセット時刻・閾値超過を機械可読JSONで返す(週次枠は見ない)。状態ファイルがまだ無い/読めない場合は `action: unknown` を返す(推定によるごまかしはしない)。
- **ヒント用 hook** `scripts/ratelimit-hint-hook.sh` — `PreToolUse` に登録(setup.sh が追加)。ツール実行前に `check-rate-limit.sh` を自動実行し、`wrap_up_and_wait` なら `additionalContext` で「rate-limit-guard に従って切り上げ→待機せよ」と**ヒントを注入**(非ブロッキング)。閾値跨ぎの**初回のみ**注入し、切り上げ作業を邪魔しない。注入済みフラグは hook 入力の `session_id` 単位で管理するので、**並行セッションが互いの通知を抑制しない**(5時間枠の `resets_at` は全セッション共通のため、グローバル1ファイルだと1セッションのヒントが他を黙らせてしまう)。発動の確実性を上げる補助で、実際の対処(切り上げ/待機/再開)はスキル本体が担う。フックは promote するだけで、起動を強制はしない。

## セットアップ(初回のみ)

`setup.sh install` が2つを導入する(どちらも**追記方式・バックアップ付き**・`uninstall` で除去):
1. **使用率キャプチャ** — 既存の statusline スクリプト(`~/.claude/statusline.sh`)に、`rate_limits` を独自ファイルへ書き出す行を追記。**statusLine のコマンドは差し替えない**ので表示・動作は不変。
2. **ヒント用 hook** — `settings.json` の `PreToolUse` に `ratelimit-hint-hook.sh` を追加(既存フックは保持)。

**statusline.sh と settings.json を書き換えるため、実行前に必ずユーザーへ確認すること。**

```bash
bash <skill>/scripts/setup.sh status     # 現状確認(両方の導入有無)
bash <skill>/scripts/setup.sh install    # 2つを追記(各ファイルを .bak にバックアップ)
bash <skill>/scripts/setup.sh uninstall  # 両方を除去(statusline.sh は原本へ、settings.json も元へ)
```

対象 statusline は `statusLine.command` から自動判定(`RLG_STATUSLINE_FILE` で上書き可)。導入後、最初の API 応答で `~/.claude/rate-limit-guard/state.json` が生成・更新され始める。導入済みなら以降のセッションでは何もしなくてよい。

## 使い方: 長時間タスク中の継続監視

長時間の自律作業では、**節目(サブタスクの完了ごと)** で必ずチェックを挟む。`/loop` で回しているならその各イテレーションで、そうでなければ作業の区切りごとに実行する(待機・自動再開は `/loop` でなくても動く。下記参照):

```bash
bash <skill>/scripts/check-rate-limit.sh
```

返ってくる JSON の `action` で分岐する:

| `action` | 意味 | やること |
|---|---|---|
| `continue` | 5時間枠が閾値未満 | そのまま作業を続行。次の節目で再チェック |
| `wrap_up_and_wait` | 5時間枠が閾値超過 | 切り上げ → リセット(数時間内)まで待機 → 自動再開 |
| `unknown` | 取得失敗 | ユーザーに状況を伝え、続行可否を確認 |

`windows.five_hour.resets_in_seconds` がリセットまでの秒数。`source` は `statusline`(実値)か `none`(取得不可)。`fresh` が値の鮮度。`note` に注意書き。

頻繁に呼びすぎない。閾値まで余裕がある(例: 70%未満)うちは数イテレーションに一度で十分。80%を超えたら毎節目で確認する。

## `wrap_up_and_wait`(5時間枠) のときの手順

**順序が命。** 待つ前に必ず「いつ中断されても損しない状態」を作る。

### 1. 切りの良いところまで進める

- いま着手中の最小単位だけ完了させ、**それ以上の新しい着手はしない**。
- リポジトリを安全な状態にする: ビルド/テストが通る状態にし、半端な編集を残さない。可能なら **WIPコミット**(例: `git add -A && git commit -m "WIP: rate-limit-guard 待機前の保存"`)。
- 再開用メモを `~/.claude/rate-limit-guard/checkpoint.md` に書く: 何のタスクか / 完了したこと / 次にやること / 触ったファイル / 再開コマンド。

### 2. 待つ(自動再開の前提)

`resets_in_seconds`(= R)だけ待つ。**リセット判定は使用率の低下ではなく、現在時刻が `resets_at` を超えたかで行う**(待機中は API 呼び出しが無く使用率が更新されないため)。

自動再開には **セッションが開いたまま動き続けていること** が前提(セッションを閉じるとローカルの待機プロセスも止まる)。開いたままなら `/loop` でなくても自動で待って再開できる:

1. **背景待機 + 自動再開(推奨・通常セッションで動く)** — `Bash` を `run_in_background: true` で実行し、リセット時刻まで待つ until ループを回す。終了するとツールが自動で再起動をかけるので、そこで再開する。前景 `sleep` はブロックされるが**背景なら動く**(実測で確認済み)。

   ```bash
   target=$(( RESETS_AT + 60 ))   # RESETS_AT = check結果の resets_at
   until [ "$(date +%s)" -ge "$target" ]; do sleep 30; done
   echo "RESET_REACHED"
   ```

   起こされたら 3.(再開)へ。**まだ `resets_at` に達していなければ**(背景待機が途中で打ち切られた等)、同じ背景待機を**張り直す**。これで待機が長くても確実に到達まで粘れる。

2. **Monitor で待つ(進捗表示・長時間向けの代替)** — 残り時間のハートビートを見たい、または待機が長い場合は `Monitor` を使う。until ループ内で残り分数を出力し、到達で `RESET_REACHED` を出して終了。`timeout_ms` は最大1時間、それより長くなら `persistent: true`。

3. **`/loop` 自走モードなら `ScheduleWakeup` でも可** — ループで回しているなら `ScheduleWakeup(min(R + 60, 3600))` で次の起床を刻む(R>3600は1時間刻み)。

起床ごとにプロンプトキャッシュ再構築(約1.25×)がかかる。背景待機を1本で張れれば再構築は実質1回で済む。詳細は `references/protocol.md` を参照。

### 3. 再開

起床したら **まず `check-rate-limit.sh` を再実行**。`action` が `continue` に戻っていれば、`checkpoint.md` を読んで作業を再開する。まだ超過していれば再度待機をスケジュール。

## フォールバックとエッジケース

- **`fresh: false`** — 状態ファイルが古い。待機中の無API期間なら正常(リセット時刻は有効)。それ以外で古いなら、短く待って(数十秒)から再チェックして最新値を取り直す。
- **`action: "unknown"`** — 状態ファイルがまだ無い/読めず、使用量を取得できない。勝手に止めも進めもせず、状況を伝えてユーザーの判断を仰ぐ(必要なら `setup.sh status` でキャプチャ導入を確認、または最初の statusLine 更新を待つ)。
- **閾値の変更** — `RL_THRESHOLD`(既定90)で調整可能。例: 余裕を持ちたいなら `RL_THRESHOLD=85 bash check-rate-limit.sh`。
- ユーザーが「上限まで使い切ってでも進めて」と明示した場合は、このスキルの自動切り上げより**ユーザーの指示を優先**する。
