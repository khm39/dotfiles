# Git 命名規約

ブランチ・コミット・タグ・PR の命名規約をまとめる。CLAUDE.md から参照される。

## ブランチ名

- タスクを反映した短い kebab-case にする（例: `fix-login-bug`, `add-worktree-rule`）
- 種別を示す場合は接頭辞を付ける: `feat/`, `fix/`, `refactor/`, `docs/`, `chore/`, `test/`（例: `fix/login-redirect`）
- worktree のディレクトリ名とブランチ名は別物。push するブランチ名には `worktree` 等の作業環境由来の接頭辞を付けない
- 一時的な作業ブランチでも、push・PR 化する可能性があるならタスク名にしておく
- 日付や個人名ではなく「何をするか」を表す名前にする

## コミットメッセージ

- 日本語で書く
- 1行目（サマリ）は `領域: 要約` の形式を推奨（例: `settings: themeをautoに追加`, `worktree運用: ブランチ命名を明記`）。領域が単一で自明なら省略してよい
- サマリは何をしたかが1行で分かるように。体言止め・命令形どちらでもよい
- 詳細が必要なら本文に箇条書きで「なぜ・何を」を書く。サマリと本文は空行で区切る
- 論理的にまとまった単位（機能追加・バグ修正・リファクタリング等）ごとにコミットする

Conventional Commits の英語 type（`feat:` 等）は必須にしない。日本語サマリ + 任意の領域プレフィックスを基本とする。

## タグ

- バージョンタグは `v` + セマンティックバージョニング（例: `v1.2.0`）
- プレリリースは `v1.2.0-rc.1`, `v1.2.0-beta.1` のように suffix を付ける

## PR タイトル

- コミットサマリと同じ方針。日本語で `領域: 要約`
- 関連する issue/チケットがあれば本文に記載する（`Closes #123` 等）
