# Rate Limit Guard

長時間の作業中に Claude のサブスク使用量を監視し、上限(既定90%)に近づいたら**作業を壊さずに切り上げ → リセットまで待機 → 自動再開**(リセットが遠い週次枠なら**報告して停止**)するためのスキル。

「途中でいきなり制限に当たって中途半端な状態で止まる」事故を防ぐのが目的。だから *いつ止まっても安全な状態を作ってから* 待つ/止まる、という順序が最重要。

## 対象範囲

**5時間枠(セッション枠)と週次(7日)枠の両方を見る。** ただし扱いが違う:

- **5時間枠**: リセットは必ず数時間以内 → 超過したら切り上げて**自動待機・自動再開**。
- **週次枠**: リセットは最大数日先。リセットが自動待機の上限(`RL_MAX_WAIT_SECONDS`、既定6時間)以内なら5時間枠と同様に待つが、それより遠ければ何時間待っても再開できないので、チェックポイントを残して**ユーザーに報告して停止**する(`wrap_up_and_stop`)。

## 仕組み(独立性)

使用量の**真の値**はサブスクの使用率で、これは statusLine に流れてくる `rate_limits` ストリームからしか取れない。本スキルは既存の statusline スクリプトに**小さなキャプチャ行を追記**して独立した状態ファイルを生成する。**statusLine のコマンド自体は差し替えない**(表示・動作は不変)。生成するファイルは本スキル専用で、外部ツール(vibe-island 等)の出力には依存しない。

- **状態キャプチャ(追記)** — `scripts/setup.sh` が既存の `~/.claude/statusline.sh` に、`rate_limits` を独自ファイル `~/.claude/rate-limit-guard/state.json` へ書き出すマーカー付きブロックを**追記**する。既存スクリプトが stdin を読む変数名は自動検出し、検出できなければ中断する(黙って動かないブロックを足さない)。`uninstall` で除去でき、往復で原状復帰する(末尾改行が無かった場合のみ補われることがある)。
- **チェッカー** `scripts/check-rate-limit.sh` — 上記の状態ファイルを読み、両枠の使用率・リセット時刻・閾値超過を機械可読JSONで返す。状態ファイルが無い/壊れている/値の型が想定外/jq不在、いずれの場合も必ず `action: unknown` のJSONを返す(推定によるごまかしはしない)。**リセット到達の誤判定もここで吸収する**: 現在時刻が `resets_at` を過ぎていれば古い使用率は前の枠の値なので超過扱いしない(`window_elapsed: true`)。待機明けは単純に再実行すれば正しい判定が返る。
- **ヒント用 hook** `scripts/ratelimit-hint-hook.sh` — `PreToolUse` に登録(setup.sh が追加。matcher は `Edit|Write|Bash|SendMessage|Agent` で、実装しないチームリードの調整作業にも反応する)。ツール実行前に `check-rate-limit.sh` を自動実行し、`approaching`/`wrap_up_and_wait`/`wrap_up_and_stop` なら `additionalContext` で対処を**ヒントとして注入**(非ブロッキング)。同じ「action+リセット時刻」の組につき**初回のみ**注入し、切り上げ作業を邪魔しない(エスカレートしたら再注入)。注入済みフラグは hook 入力の `session_id` 単位で管理するので、**並行セッションやチームメンバーが互いの通知を抑制しない**。発動の確実性を上げる補助で、実際の対処(切り上げ/待機/再開)はスキル本体が担う。フックは promote するだけで、起動を強制はしない。

## セットアップ(初回のみ)

`setup.sh install` が2つを導入する(どちらも**追記方式・バックアップ付き**・`uninstall` で除去):
1. **使用率キャプチャ** — 既存の statusline スクリプト(`~/.claude/statusline.sh`)に、`rate_limits` を独自ファイルへ書き出す行を追記。**statusLine のコマンドは差し替えない**ので表示・動作は不変。
2. **ヒント用 hook** — `settings.json` の `PreToolUse` に `ratelimit-hint-hook.sh` を追加(既存フックは保持)。

**statusline.sh と settings.json を書き換えるため、実行前に必ずユーザーへ確認すること。**

```bash
bash <skill>/scripts/setup.sh status     # 現状確認(両方の導入有無、state.json の生成有無)
bash <skill>/scripts/setup.sh install    # 2つを追記(各ファイルを .bak にバックアップ)
bash <skill>/scripts/setup.sh uninstall  # 両方を除去(statusline.sh は原本へ、settings.json も元へ)
```

対象 statusline は `statusLine.command` から自動判定(`RLG_STATUSLINE_FILE` で上書き可)。**install 後は、最初の statusline 更新(API応答)を待ってから `setup.sh status` で `state.json` の生成を必ず確認する** — キャプチャは既存スクリプトの構造に依存するため、生成されなければ導入は成功していない。導入済みなら以降のセッションでは何もしなくてよい。なお hook にはスキルの絶対パスが登録されるため、**worktree 内のスキルから install してはいけない**(setup.sh が検知して中断する。マージ後の本体パスから実行する)。

## 使い方: 長時間タスク中の継続監視

長時間の自律作業では、**節目(サブタスクの完了ごと)** で必ずチェックを挟む。`/loop` で回しているならその各イテレーションで、そうでなければ作業の区切りごとに実行する(待機・自動再開は `/loop` でなくても動く。下記参照):

```bash
bash <skill>/scripts/check-rate-limit.sh
```

返ってくる JSON の `action` で分岐する:

| `action` | 意味 | やること |
|---|---|---|
| `continue` | 両枠とも余裕あり | そのまま作業を続行。数イテレーションに一度の再チェックで十分 |
| `approaching` | いずれかの枠が警告域(既定80%以上・閾値未満) | 続行してよいが、**以後は毎節目でチェック**し、切りの良い単位で進める |
| `wrap_up_and_wait` | 5時間枠が超過、または週次枠が超過でリセットが近い | 切り上げ → `wait_until` まで待機 → 自動再開 |
| `wrap_up_and_stop` | 週次枠が超過でリセットが自動待機の上限より遠い | 切り上げ → ユーザーに報告して**停止**(自動待機しない) |
| `unknown` | 取得失敗 | ユーザーに状況を伝え、続行可否を確認 |

主なフィールド: `limiting_window`(判定の原因になった枠)、`wait_until`(待機の目標エポック秒。両枠超過なら遅い方)、`windows.<枠>.resets_in_seconds`(リセットまでの秒数)、`windows.<枠>.window_elapsed`(リセット到達済みで使用率が前の枠の値)、`source`(`statusline`=実値 / `none`=取得不可)、`fresh`(値の鮮度)、`note`(注意書き)。

## `wrap_up_and_wait` のときの手順

**順序が命。** 待つ前に必ず「いつ中断されても損しない状態」を作る。

### 1. 切りの良いところまで進める

- いま着手中の最小単位だけ完了させ、**それ以上の新しい着手はしない**。
- リポジトリを安全な状態にする: ビルド/テストが通る状態にし、半端な編集を残さない。可能なら **WIPコミット**(例: `git add -A && git commit -m "WIP: rate-limit-guard 待機前の保存"`)。
- 再開用メモを `~/.claude/rate-limit-guard/checkpoints/${CLAUDE_CODE_SESSION_ID}.md` に書く(**セッションごとに別ファイル**。並行セッションが互いのチェックポイントを上書きしないため。`CLAUDE_CODE_SESSION_ID` は Bash 環境変数として参照できる): 何のタスクか / 完了したこと / 次にやること / 触ったファイル / 再開コマンド。書式と未設定時のフォールバックは `references/protocol.md` を参照。
- **エージェントチームのリードとして作業中なら、待機前にメンバーも切り上げさせる**(下記「チームで作業中の場合」)。メンバーを走らせたまま待機すると、その間もメンバーが同じサブスク枠を消費し続けて本末転倒になる。

### 2. 待つ(自動再開の前提)

チェック結果の `wait_until`(エポック秒)まで待つ。**リセット判定は使用率の低下ではなく時刻で行う**(待機中は API 呼び出しが無く使用率が更新されないため)。

自動再開には **セッションが開いたまま動き続けていること** が前提(セッションを閉じるとローカルの待機プロセスも止まる)。推奨手段は**背景待機1本**: `Bash` を `run_in_background: true` で実行し、`wait_until` まで待つ until ループを回す。終了するとツールが自動で再起動をかけるので、そこで再開する(前景 `sleep` はブロックされるが背景なら動く。実測で確認済み)。

```bash
target=$(( WAIT_UNTIL + 60 ))   # WAIT_UNTIL = check結果の wait_until
until [ "$(date +%s)" -ge "$target" ]; do sleep 30; done
echo "RESET_REACHED"
```

起こされたら 3.(再開)へ。まだ `wait_until` 未到達なら同じ背景待機を張り直す。代替手段(進捗表示つきの Monitor / `/loop` 自走時の ScheduleWakeup)、セッションを閉じる場合のフォールバック、キャッシュ再構築コストの考慮は `references/protocol.md` の「待機の仕方」を参照。

### 3. 再開

起床したら **まず `check-rate-limit.sh` を再実行**。リセット到達済みならチェッカーが `window_elapsed: true` として超過扱いを解除するので、`action` が `continue` 系に戻っていることを確認し、自セッションのチェックポイント(`checkpoints/${CLAUDE_CODE_SESSION_ID}.md`)を読んで作業を再開する。チームを切り上げさせていた場合はメンバーに作業を再投入する。まだ超過していれば再度待機をスケジュール。

## `wrap_up_and_stop`(週次枠・リセットが遠い) のときの手順

何時間待っても枠は戻らないので、自動待機はしない。代わりに:

1. 上記「1. 切りの良いところまで進める」と同じ手順で安全な状態を作る(チーム作業中はメンバーも切り上げさせる)。
2. ユーザーに報告する: 週次枠の使用率 / リセット時刻(`resets_at` を人間可読に変換) / ここまでの進捗 / チェックポイントファイルのパス。`PushNotification` が使えるなら通知も送る。
3. **停止する**。勝手に「少しだけ続行」しない。再開はリセット後にユーザー(または次のセッション)がチェックポイントから行う(別セッションからの探し方は `references/protocol.md`)。

## チームで作業中の場合(リード向け)

自分がエージェントチームのリードで、`wrap_up_and_wait` / `wrap_up_and_stop` になったら、**自分が待機・停止する前にメンバーを穏やかに切り上げさせる**。メンバーも同じサブスク枠を消費するため、放置すると待機の意味がなくなる。順序は: ①全メンバーへ SendMessage で一斉に切り上げ指示 → ②報告を回収 → ③チーム状態表つきチェックポイントを書く → ④自分の待機/停止 → ⑤再開時に残作業を再投入。指示の定型文・報告待ちのタイムアウト・再投入の詳細は `references/protocol.md` の「チーム切り上げの詳細」を参照。

待機・再開の主導権はリードに一本化する。メンバーは自分の判断で長時間待機せず、切り上げて報告するところまでで止まるのが正しい挙動。なおリード自身は実装ツールをあまり使わないため hook のヒントに頼りすぎない — メンバーからの報告受信や調整の節目ごとに自分でチェックを挟むこと。

## フォールバックとエッジケース

- **`fresh: false`** — 状態ファイルが古い。待機中の無API期間なら正常(リセット時刻は有効)。それ以外で古いなら、短く待って(数十秒)から再チェックして最新値を取り直す。
- **`action: "unknown"`** — 使用量を取得できない(状態ファイルが無い/壊れている/jq不在)、**または超過しているのに `resets_at` が取れず待機目標を計算できない**(`note` に理由が入る)。いずれも勝手に止めも進めもせず、状況を伝えてユーザーの判断を仰ぐ(必要なら `setup.sh status` でキャプチャ導入を確認、または最初の statusLine 更新を待つ)。
- **閾値・待機上限の変更** — `RL_THRESHOLD`(既定90)/ `RL_WARN_THRESHOLD`(既定80)/ `RL_MAX_WAIT_SECONDS`(既定21600=6時間)で調整可能。例: 余裕を持ちたいなら `RL_THRESHOLD=85 bash check-rate-limit.sh`。
- ユーザーが「上限まで使い切ってでも進めて」と明示した場合は、このスキルの自動切り上げより**ユーザーの指示を優先**する。
